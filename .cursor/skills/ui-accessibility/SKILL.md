# Accessibility Skill

## Overview

Every component and page generated by the `/ui` command must meet WCAG 2.1 AA compliance. This skill provides the implementation patterns, validation checklist, and audit process for ensuring all UI output is accessible to users with disabilities, including those using screen readers, keyboard-only navigation, or high-contrast displays.

## Prerequisites

- `UI_KNOWLEDGE.md` §4 loaded (common mistakes including missing aria)
- Project uses semantic HTML5 elements (verified during setup)

## Steps

### Step 1 — Semantic HTML Structure

Use the correct HTML5 semantic elements. NEVER use `<div>` or `<span>` for structural elements:

| Purpose | Correct Element | Wrong Element |
|---------|----------------|--------------|
| Site header | `<header>` | `<div class="header">` |
| Site navigation | `<nav aria-label="Main">` | `<div class="nav">` |
| Main content | `<main>` | `<div id="content">` |
| Page section | `<section aria-labelledby="section-heading">` | `<div class="section">` |
| Supporting content | `<aside aria-label="Related">` | `<div class="sidebar">` |
| Blog post | `<article>` | `<div class="post">` |
| Site footer | `<footer>` | `<div class="footer">` |
| Grouped form fields | `<fieldset><legend>` | `<div class="group">` |

**Success Criteria:** Zero structural `<div>` elements where a semantic element is appropriate.

---

### Step 2 — ARIA Attributes Reference

Apply the correct ARIA for each pattern:

**Buttons and Links:**
```tsx
// Buttons with icon only — MUST have aria-label
<button aria-label="Close dialog" onClick={onClose}>
  <XMarkIcon className="h-5 w-5" aria-hidden="true" />
</button>

// Toggle buttons — must communicate state
<button aria-pressed={isActive} onClick={toggle}>
  {isActive ? 'On' : 'Off'}
</button>

// Buttons that expand content
<button aria-expanded={isOpen} aria-controls="menu-items" onClick={toggle}>
  Menu
</button>
```

**Navigation:**
```tsx
<nav aria-label="Main navigation">
  <ul role="list">
    <li>
      <a href="/about" aria-current={isCurrentPage ? 'page' : undefined}>
        About
      </a>
    </li>
  </ul>
</nav>

// Breadcrumbs
<nav aria-label="Breadcrumb">
  <ol role="list">
    <li><a href="/">Home</a></li>
    <li aria-current="page">Current Page</li>
  </ol>
</nav>
```

**Forms:**
```tsx
// All inputs MUST have a label
<div>
  <label htmlFor="email" className="block text-sm font-medium text-gray-700 dark:text-gray-300">
    Email address
  </label>
  <input
    id="email"
    type="email"
    name="email"
    autoComplete="email"
    required
    aria-required="true"
    aria-describedby="email-error"
    className="mt-1 block w-full rounded-md border-gray-300 ..."
  />
  <p id="email-error" role="alert" className="mt-2 text-sm text-red-600">
    {errorMessage}
  </p>
</div>
```

**Modals/Dialogs (using Headless UI):**
```tsx
// Headless UI Dialog handles focus trap and aria-modal automatically
<Dialog open={isOpen} onClose={setIsOpen}>
  <DialogTitle>Confirm action</DialogTitle>
  {/* Headless UI sets role="dialog" and aria-modal="true" */}
</Dialog>
```

**Live Regions (dynamic content):**
```tsx
// Notifications/toasts
<div role="status" aria-live="polite" aria-atomic="true">
  {notification}
</div>

// Error alerts
<div role="alert" aria-live="assertive">
  {errorMessage}
</div>
```

**Success Criteria:** All interactive elements have appropriate ARIA attributes.

### Step 3 — Keyboard Navigation

Every interactive element MUST be keyboard accessible:

| Element | Keyboard behavior |
|---------|------------------|
| Button | Enter / Space activates |
| Link | Enter activates |
| Modal | Esc closes, Tab cycles focus within |
| Dropdown menu | Enter/Space opens, Arrow keys navigate, Esc closes |
| Tabs | Arrow keys cycle between tabs |
| Accordion | Enter/Space toggles panel |
| Form inputs | Tab moves between fields |

**Focus management for modals:**
```tsx
// Headless UI handles focus trap automatically
// For custom implementations:
useEffect(() => {
  if (isOpen) {
    // Focus the first focusable element
    firstFocusableRef.current?.focus()
  }
}, [isOpen])
```

**Success Criteria:** All components verified keyboard-operable without mouse.

---

### Step 4 — Focus Styles

NEVER remove focus outlines. Use Tailwind's visible focus classes:

```tsx
// Standard focus style — ALWAYS include
className="focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-offset-2 focus-visible:ring-blue-500"

// For dark backgrounds
className="focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-white"

// Do NOT use:
className="focus:outline-none" // removes focus without replacement — WCAG violation
```

**Success Criteria:** Every interactive element has a visible focus indicator.

---

### Step 5 — Color Contrast

Minimum contrast ratios (WCAG 2.1 AA):
- Normal text (< 18pt): **4.5:1**
- Large text (≥ 18pt bold or 24pt): **3:1**
- UI components (borders, icons): **3:1**

Tailwind CSS pairs that meet AA:
| Text | Background | Ratio |
|------|-----------|-------|
| `text-gray-900` | `bg-white` | 21:1 ✅ |
| `text-gray-700` | `bg-white` | 10:1 ✅ |
| `text-gray-600` | `bg-white` | 5.9:1 ✅ |
| `text-white` | `bg-blue-600` | 4.6:1 ✅ |
| `text-white` | `bg-gray-700` | 10:1 ✅ |
| `text-gray-400` | `bg-white` | 2.6:1 ❌ (not AA compliant for body) |

**Rule:** `text-gray-400 on bg-white` is ONLY acceptable for placeholder or decorative text. Never for meaningful content.

**Success Criteria:** All text/background combinations meet 4.5:1 (normal) or 3:1 (large text).

### Step 6 — Screen Reader Support

```tsx
// Visually hidden text for screen readers (sr-only class)
<button>
  <span className="sr-only">Delete item named</span>
  {itemName}
  <TrashIcon className="h-4 w-4" aria-hidden="true" />
</button>

// Skip link (place at top of layout)
<a
  href="#main-content"
  className="sr-only focus:not-sr-only focus:absolute focus:top-4 focus:left-4 focus:z-50 focus:px-4 focus:py-2 focus:bg-blue-600 focus:text-white focus:rounded"
>
  Skip to main content
</a>

// Decorative images
<img src="decoration.jpg" alt="" aria-hidden="true" />

// Meaningful images
<img src="team-photo.jpg" alt="The Acme team at our annual offsite in 2024" />
```

**Success Criteria:** All meaningful content readable by screen readers; all decorative elements hidden.

### Step 7 — A11y Audit Checklist

When running `/ui audit` a11y check, scan each file for:

| Check | Severity | Pattern |
|-------|----------|---------|
| Images missing alt | Critical | `<img` without `alt=` |
| Buttons missing label | Critical | Icon-only buttons without `aria-label` |
| Missing form labels | Critical | `<input` without associated `<label>` |
| `focus:outline-none` without ring | High | Removed focus indicator |
| Non-semantic structure | Medium | `<div onclick=...>` instead of `<button>` |
| Missing lang attribute | Medium | `<html>` without `lang="en"` |
| Insufficient contrast | High | `text-gray-400 on bg-white` for content |
| Missing skip link | Low | No skip-to-main-content link |

Report format:
```
## A11y Audit: [filename]
| Issue | Severity | Line | Fix |
| :--- | :--- | :--- | :--- |
| <img> missing alt attribute | Critical | 24 | Add alt="[description]" |
```

## Verification

- [ ] All HTML elements are semantic (no structural divs)
- [ ] All images have `alt` attributes (descriptive or empty for decorative)
- [ ] All interactive elements keyboard accessible
- [ ] All icon-only buttons have `aria-label`
- [ ] All form inputs have associated `<label>` with matching `id`/`htmlFor`
- [ ] Focus styles present (`focus-visible:ring-2`) — `focus:outline-none` never used alone
- [ ] Color contrast meets 4.5:1 for body text, 3:1 for large text/UI
- [ ] Live regions used for dynamic content
- [ ] `lang="en"` (or correct locale) on `<html>` element
